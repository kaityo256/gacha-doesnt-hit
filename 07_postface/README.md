# おわりに

本書ではリボ払いの返済シミュレーションに始まり、ガチャや、ゾンビパニック、渋滞、マイノリティ・ゲームまで、様々な現象をシミュレーションで再現してきました。数値シミュレーションでは、プログラムを書いて、それを実行して何が起きるかを調べます。その意味では実験に近い性質を持っていますが、数値シミュレーションではプログラムを書いたのは自分ですから、そのプログラムが表す世界のルールを全て知っていることになります。しかし、たとえ自分が書いたプログラムであっても、そのプログラムの結果を予想できるわけではありません。バクテリアの増殖シミュレーションのように、ルールが簡単であっても、パラメータの値によっては予想外の結果が起きたりします。また、シミュレーションの結果、望む結果を再現できなかった時に、どのようにルールを変えれば良いかを考えるのも容易ではありません。このように、数値シミュレーションのプログラマーは、そのシミュレーションの世界の全てのルールを知る「全知」の神のような存在でありながら、「全能」ではありません。これが数値シミュレーションの面白いところです。数値シミュレーションでは、私たちを取り巻く複雑な世界をなるべく簡単なルールでモデル化し、その簡単なルールの世界が豊かで複雑な現象を引き起こすことを調べ、その理由を考え、また別のルールで何が起きるかを調べていきます。このプロセスを繰り返すことで研究が進み、世界を理解しようと試みるのが数値シミュレーションです。

私の専門は、特にスーパーコンピューターを使った大規模数値シミュレーションです。スーパーコンピュータを使う時には、プログラムを書いて、それをスーパーコンピュータに実行予約をします。スーパーコンピュータはみんなが使うものですから、自分のプログラムはすぐには実行されません。自分のプログラムの実行順が来ると、メールで通知がくるので、早速スーパーコンピュータにログインして結果をチェックします。プログラムは実行しながら結果をファイルに吐き出していくので、それを順次別のプログラムを使って可視化します。そして、思うような結果が出た、出なかったと一喜一憂するのです。

少し前のことになりますが、当時日本で一番大きいスーパーコンピュータであった京コンピューターを使ったことがあります。京コンピューターは88128個のCPUからなる巨大なシステムで、普段は研究者達が分割して使います。しかし申請が認められた人は、1ヶ月に一度だけ、京コンピューターの全てを使ってシミュレーションするチャンスがありました。私の申請が採択され、京コンピューターの全系計算ができることになりました。当時の私の研究テーマは沸騰現象であり、京コンピューター全てを使って水が沸騰するシミュレーションを実施する計画でした。十分準備をしたつもりでしたが、私はこの全系計算に二度失敗し、最後のチャンスとなる三回目の計算に挑みます。深夜、プログラムが無事に走り始めたという連絡がメールで届き、そのままスーパーコンピュータにログインして待っていると、プログラムが巨大なファイルを吐き出し始めます。そのファイルを、また別のスーパーコンピューターに転送して解析すると、ちゃんと泡が出始めていることがわかりました。その当時の興奮は今でも覚えています。

「はじめに」において、大学ではみんな違うことを学ぶのに、なぜ共通の「資格」を得るのでしょう、という話を書きました。大学の研究活動では、理論でも実験でも数値シミュレーションでも、まずは先生から言われたことを実施することになるでしょう。しかし、研究の結果起きることは先生にもわかりません。出てきた結果を見て、先生と一緒に理由を考え、次に何をすべきか考え、また次の結果を得る、そのサイクルを回すことが研究であり、そのサイクルを回す力こそ大学で学ぶべきことです。

本書で紹介したシミュレーションは、どれも小さなルールから出発しました。しかし、その先に現れる世界は、しばしば私たちの予想を超えていきます。だからこそ、プログラムを書き、動かし、結果を眺め、理由を考えることには大きな意味があります。皆さんもぜひ、自分だけの小さな世界をコンピュータの中につくり、そこで何が起きるのかを確かめてみてください。その驚きと発見の先に、研究の楽しさがあります。
